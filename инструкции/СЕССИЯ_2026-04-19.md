## TL;DR

За сессию закрыли 4 крупных пакета:

1. **Keyword-gated LLM triggers** — бот отвечает не только на @-mention /
   reply / commands, но и когда в тексте / голосовом транскрипте нашёлся
   keyword из таблицы `trigger_keywords`.
2. **Роутинг-баг**: `mentions.router` тенил `messages.router`. Фиксен
   фильтр-классом `AddressedToMe`.
3. **Whisper Latin-bias**: модель транслитерировала «Алкаш»→«Alkaz»,
   `initial_prompt` + чистка ASCII из подсказки + latin-fallback keywords.
4. **Стикеры**: полный send+learn pipeline. Бот собирает паки, логирует
   контекст использования команды, Claude может отвечать стикером через
   поле `sticker_emoji`.

Плюс инфраструктура: доступ к Railway GraphQL из Claude через Keychain,
capability-matrix в system prompt, документация.

**Задеплоенный SHA на момент закрытия сессии:** `c396a10` (feat stickers).

## 1. Задача, с которой всё началось

Юзер пожаловался что бот не реагирует на сообщения со словом «бот» в
группе, хотя должен (keyword matcher должен ловить).

Пример: *«Бот, а бот. Ты как»* — молчок.

Контекст проекта — это Telegram-бот для учёта операций в Сбер-
кабинетах (POA-withdrawals, обмен RUB→USDT, аренда кабинетов), крутится
в одной группе с двумя whitelisted-юзерами (Казах + Арбуз).

## 2. Root cause — router-shadow bug

`mentions.router` был декорирован:

```python
@router.message(F.text | F.caption)
async def on_mention(message: Message) -> None:
    if not await _addressed_to_me(message):
        return
    ...
```

В aiogram 3 первый роутер, чей *фильтр* матчит, выигрывает — `_addressed_to_me`
отрабатывал УЖЕ после того как роутер захапал сообщение. Результат:
любой текст уходил в `mentions`, inline-проверка возвращала False,
routing останавливался. `messages.on_message` с keyword-gate никогда
не видел ни одного обычного сообщения.

**Фикс (коммит `e7bbc98`):**

- Создан `src/bot/filters/addressed.py` → класс `AddressedToMe(Filter)`,
  возвращающий тот же boolean что и старая inline-проверка.
- `mentions.py`: `@router.message(AddressedToMe())` вместо `F.text | F.caption`.
- Inline-хелпер `_addressed_to_me` удалён как мёртвый.

Верификация: в `message_log` видно что сразу после деплоя бот отвечал
на «Бот, а бот. Ты как» через ~4 секунды (keyword «бот» сматчился,
вошёл в `messages.on_message`, сработал flush).

## 3. Whisper Latin-bias — голосовые не триггерили LLM

Проблема: `faster-whisper` (модель `small`, int8) мис-транскрибил:

- «Бот» → «Бод» / «Вот»
- «Алкаш» → «Alkaz» (латиница!)
- «Цифровой пидорас» → «Цифровой тихой соц»

Keyword matcher (substring, case-insensitive) эти мутации не ловил, бот
молчал на голосовые с явно произнесёнными keyword-ами.

### Три слоя фикса (коммиты `375686a` + `e9ac25f`)

**Слой 1 — `initial_prompt` для Whisper.**
faster-whisper принимает `initial_prompt`: «prior context which the model
has already seen». Туда пихаем список keywords — модель биасится к
нашему словарю. Но первая итерация включала ASCII (`al_kazbot`) и это
усилило латинизацию.

**Слой 2 — фильтр Cyrillic-only для prompt.**
`src/core/voice_transcribe._is_cyrillic_word(s)` — True только если
строка содержит кириллицу И не содержит латинских букв. В
`_build_whisper_prompt` прогоняются ТОЛЬКО они. ASCII keywords вроде
`al_kazbot` остаются в БД для текстового matching, но в подсказке
Whisper не появляются.

**Слой 3 — Latin-варианты как keywords-подстраховки.**
Даже с cyrillic-only подсказкой Whisper иногда всё равно выдаст
латиницу. Миграция `1033b73e2805` сидит `alkaz`, `alkash`, `erzhan`
в `trigger_keywords` — substring-матчер их подловит.

## 4. Capability matrix — Claude перестаёт врать про возможности

Юзер заметил что бот говорил *«не умею менять аватарку, у меня нет
метода setChatPhoto в арсенале»* — прямое враньё, т.к.
`/avatar` команда реально работает (`src/bot/handlers/commands.py:768`,
`bot.set_chat_photo(...)` вызывается). Claude не знал о slash-commands
потому что system prompt их не перечислял.

**Фикс в `src/llm/system_prompt.py` (коммит `e9ac25f`):**

Добавлен блок `# Твои реальные капабилити (капабилити-матрица)` с тремя
разделами:

1. **Что УМЕЕШЬ сам** — парсинг операций, Whisper voice, Vision PDF/фото,
   запись в knowledge base.
2. **Что может юзер через slash-commands** (и бот должен подсказать):
   `/avatar`, `/keywords`, `/report`, `/balance`, `/stock`, `/fx`, `/partners`,
   `/history`, `/undo`, `/silent`, `/voices`, `/resync`, `/feedback`.
3. **Что НЕ умеешь**: отправка фото/видео/гифки/голосовых, смена
   собственного имени/аватарки, выход за пределы чата.

Плюс прямое указание: *«На "можешь Х?" отвечай `да, через /X` или
`нет`. Никаких отговорок про "я бухгалтер, а не дизайнер"»*.

В следующем коммите (`c396a10`) matrix обновлена — стикеры переехали
из «не умею» в «умею через `sticker_emoji`».

## 5. Стикеры — полный pipeline (коммит `c396a10`)

### Data layer

Новая таблица `sticker_usage` (migration `486b28e0b462`):

```sql
CREATE TABLE sticker_usage (
    id SERIAL PRIMARY KEY,
    sticker_file_unique_id TEXT,
    sticker_set TEXT,
    emoji TEXT,
    tg_user_id BIGINT,
    chat_id BIGINT,
    tg_message_id BIGINT,
    preceding_text TEXT,        -- последние ~3 сообщения до стикера
    sent_by_bot BOOLEAN DEFAULT false,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

Плюс индексы на file_unique_id / set / emoji / tg_user_id / chat_id /
created_at.

Модель: `src/db/models.py::StickerUsage`.

### Ingest

`src/bot/handlers/stickers.py::on_sticker`:

1. Upsert стикера в `seen_stickers` (было раньше).
2. Если пак известен — `bot.get_sticker_set(...)` и upsert всех
   siblings (было раньше).
3. **НОВОЕ**: pull last 3 non-empty `message_log` записей до этого
   стикера → кладём в `preceding_text` → `sticker_repo.log_usage(sent_by_bot=False)`.

Это «learning channel»: когда Claude решит стикер отправить, он
увидит в system prompt — кто какой стикер посылает в каком контексте.

### Send path

`src/core/batch_processor.py::_maybe_send_sticker`:

- Если `analysis.sticker_emoji` заполнен → `sticker_repo.pick_by_emoji([emoji])`.
- Fallback: стрипает ZWJ / variation selectors, пробует ещё раз.
- Если нашёлся — `bot.send_sticker(chat_id, file_id, reply_to)`,
  `bump_usage`, `log_usage(sent_by_bot=True)`, `log_bot_reply`.
- Если не нашёлся — silent no-op (юзер даже не поймёт что бот пытался).

Вставлен в `flush()` после `chat_reply` отправки. Text и стикер могут
лететь оба, или только один из них.

### LLM integration

- `BatchAnalysis.sticker_emoji: str | None` — новое поле pydantic-модели.
- `ANALYZE_TOOL.input_schema.properties.sticker_emoji` — JSON Schema
  entry с объяснением правил.
- `_collect_sticker_context()` в `batch_analyzer.py` — pull
  `pack_emoji_summary` + `recent_usage_examples` из БД перед каждым
  analyze_batch call.
- `build_system_blocks(..., sticker_pack_emojis=..., sticker_usage_examples=...)`
  добавляет новый cached block `# Стикеры` в system prompt.
- `render_sticker_context()` в `system_prompt.py` рендерит: emoji-спектр,
  per-pack списки, примеры usage (кто отправил X после «...»).

## 6. Инфраструктура — Railway GraphQL из Claude

### Проблема

Раньше у Claude не было прямого доступа к Railway. На «проверь деплой /
логи» юзер просил скинуть вручную, было трение.

Endpoint: `https://backboard.railway.com/graphql/v2`
Auth header: `Authorization: Bearer <token>`

Ключевые queries (копипасти в следующем чате):

```graphql

  name
  services { edges { node { id name } } }

# Последние деплои

  environmentId: "4ce7e1fd-5414-4c30-8505-46ce1ff0c5b7"
```

## 7. Keychain-хранилище секретов

Цель — чтобы в новом чате Claude не собирал токены с нуля.

### Скрипты

- **`scripts/secrets.sh`** — обёртка над macOS `security` (Keychain):
  - `set <name> <value>` — стандартный CLI-put
  - `set-prompt <name>` — запросит значение в терминале, не засветит в истории
  - `get <name>` — stdout plaintext
  - `list` — известные имена
  - `remove <name>` — удалить

- **`scripts/load-secrets.sh`** — `source`-friendly. Читает Keychain и
  экспортит env-vars по таблице `MAPPING`. На данный момент:
  - `railway-token` → `RAILWAY_API_TOKEN`, `RAILWAY_TOKEN`
  - `anthropic-api-key` → `ANTHROPIC_API_KEY`
  - `telegram-bot-token` → `TELEGRAM_BOT_TOKEN`
  - `prod-database-url` → `DATABASE_URL`

### В новом чате достаточно

```bash
source scripts/load-secrets.sh
```

…и у Claude в env будут токены без шаманства.

### Как это безопасно

- macOS Keychain шифрует значения ключом, который разблокируется только
  при логине пользователя.
- Даже `cat ~/Library/Keychains/login.keychain-db` даст бинарный мусор.
- Процесс, который пробует `security find-generic-password -w`, первый
  раз получит modal-диалог от macOS «Always Allow? Allow? Deny?» — юзер
  контролирует доступ.
- В git попадают только скрипты, сами секреты — нигде в репозитории.

## 10. Быстрый runbook «бот молчит»

1. `source scripts/load-secrets.sh`
2. GraphQL → проверить последний deployment: status должен быть SUCCESS.
3. `curl https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/getMe` → проверить
   что бот alive и `can_read_all_group_messages: true`.
4. DB → проверить что keywords есть в `trigger_keywords`, taблица
   `message_log` накапливает входящие (должны быть свежие записи если
   в группу пишут).
5. Логи деплоя → ищи `Update id=... is handled` — если нет, Telegram не
   доставляет апдейты.
6. Прод-DB: последний intent_detected на голосовых — проверить что
   Whisper не выдаёт мусор (сверить с исходным звуком).
